# Детальный План Реализации: Агентский RAG с Qwen3 для Анализа Пенсионных Дел

Этот документ описывает пошаговый план внедрения агентского подхода для анализа пенсионных дел с использованием Qwen3, Qwen-Agent, кастомного RAG-инструмента и графа знаний Neo4j.

## 1. Разработка Кастомного RAG-Инструмента (`PensionKnowledgeRetrieverTool`)

**Цель:** Создать инструмент для Qwen-Agent, который инкапсулирует логику поиска информации в базе знаний (LlamaIndex RAG + Neo4j).

**Место:** Новый файл `backend/app/agent_tools/pension_knowledge_retriever_tool.py`.

**Шаги:**

1.  **Определить класс `PensionKnowledgeRetrieverTool`:**
    *   Наследовать от `qwen_agent.tools.base.BaseTool`.
    *   Определить атрибуты:
        *   `name`: Строка, например, `"pension_knowledge_retriever"`.
        *   `description`: Подробное описание для LLM, что делает инструмент и какую информацию он может предоставить (например, "Извлекает и обобщает релевантные статьи пенсионного законодательства РФ и связанные с ними условия на основе поискового запроса. Может использовать информацию о типе пенсии или конкретных жизненных обстоятельствах.").
        *   `parameters`: JSON-схема, описывающая входные параметры для метода `call()`. Минимум:
            *   `query` (string, required): Поисковый запрос, сформулированный LLM.
            *   (Опционально) `pension_type` (string, optional): Конкретный тип пенсии для уточнения поиска.
            *   (Опционально) `case_context` (string, optional): Краткий контекст дела для более точного поиска.

2.  **Реализовать метод `call(self, params: str, **kwargs) -> str`:**
    *   **Парсинг `params`:** Преобразовать JSON-строку `params` в словарь (например, с помощью `json5.loads(params)`).
    *   **Взаимодействие с RAG (LlamaIndex):**
        *   Инициализировать или получить доступ к компонентам вашего RAG-пайплайна из `backend/app/rag_core/` (например, `PensionRAG` или его части). Это может потребовать небольшого рефакторинга `PensionRAG`, чтобы его методы поиска были легко вызываемы.
        *   Выполнить поиск на основе `query` (и опциональных параметров) для получения релевантных текстовых чанков.
        *   Применить реранкинг.
    *   **Обогащение из Neo4j:**
        *   Для каждого релевантного чанка или для набора ключевых сущностей, извлеченных из чанков:
            *   Сформулировать запросы к Neo4j (используя функции из `backend/app/graph_builder.py` или `graph_enricher.py`, которые, возможно, тоже потребуют адаптации).
            *   Получить связанную информацию (например, определения терминов, связанные статьи, условия, применимые к типам пенсий).
        *   Интегрировать обогащенную информацию с результатами RAG. Можно добавлять текст или структурированные данные.
    *   **Формирование результата:**
        *   Объединить и структурировать найденную и обогащенную информацию в единый текстовый блок.
        *   (Опционально, но рекомендуется) Если текста много, применить краткую суммаризацию (возможно, с помощью еще одного вызова LLM с простым промптом, но это усложнит инструмент).
        *   Вернуть результат в виде строки. Этот текст будет передан LLM-агенту.
    *   **Обработка ошибок:** Предусмотреть обработку ситуаций, когда ничего не найдено или произошли ошибки в RAG/Neo4j, и вернуть информативное сообщение.

3.  **Регистрация инструмента:**
    *   Использовать декоратор `@register_tool('pension_knowledge_retriever')` над классом.
    *   Или зарегистрировать инстанс позже при создании агента.

4.  **Начальное Тестирование:**
    *   Создать тестовые вызовы инструмента с различными запросами, чтобы убедиться в его работоспособности до интеграции с агентом.

## 2. Создание Главного Агента (`QwenPensionAgent`)

**Цель:** Создать агента, который будет управлять логикой анализа пенсионного дела, используя Qwen3 и `PensionKnowledgeRetrieverTool`.

**Место:** Новый файл `backend/app/agents/qwen_pension_agent.py`.

**Шаги:**

1.  **Определить класс `QwenPensionAgent`:**
    *   Рекомендуется наследовать от `qwen_agent.agents.Assistant` для использования встроенных возможностей планирования и вызова инструментов.
2.  **Инициализация (`__init__`):**
    *   **Конфигурация LLM (`llm_cfg`):**
        *   `model`: Указать имя модели Qwen3, доступной через Ollama (например, `qwen2:7b` или актуальное имя модели Qwen3).
        *   `model_server`: Адрес вашего локального сервера Ollama (обычно `http://localhost:11434/v1` или аналогичный, если Ollama предоставляет OpenAI-совместимый эндпоинт; если нет, Qwen-Agent может иметь прямой коннектор или потребуется адаптер). Для Qwen3 и Ollama может потребоваться кастомная настройка или использование параметра `model_type`, если Qwen-Agent поддерживает Ollama напрямую.
        *   `api_key`: Обычно 'EMPTY' или 'ollama' для локального Ollama.
        *   `generate_cfg`:
            *   `enable_thinking`: `True` или `False` (или управлять через `extra_body` как в примере `assistant_qwen3.py`, если используется OpenAI-совместимый API Ollama). Экспериментировать с этим.
            *   Другие параметры генерации (температура, top_p и т.д.).
    *   **Системный промпт (`system_message`):**
        *   Четко определить роль агента (например, "Ты — эксперт по пенсионному законодательству РФ, помогающий анализировать пенсионные дела.").
        *   Описать доступные инструменты (особенно `pension_knowledge_retriever`) и когда их использовать.
        *   Задать формат ожидаемого ответа (включая анализ, ссылки, статус, уверенность).
        *   Примеры (few-shot) того, как агент должен рассуждать и использовать инструмент, могут быть очень полезны.
    *   **Список инструментов (`function_list`):**
        *   Включить ваш зарегистрированный инструмент: `['pension_knowledge_retriever']`.
        *   Можно также включить `code_interpreter`, если агент должен выполнять какие-то расчеты или обработку данных.
    *   Передать эти параметры в конструктор `super().__init__(...)`.

3.  **Основная логика (если не используется `Assistant` напрямую, или кастомизация `_run`):**
    *   Метод `run_analysis(self, case_description_text: str, case_data_structured: CaseDataInput) -> Dict:`
        *   **Формирование входных сообщений для LLM:** Создать список сообщений (`messages`) для `Assistant.run()`. Первое сообщение может содержать системный промпт, последующие — структурированные данные дела и текстовое описание.
        *   **Вызов `Assistant.run(messages)`:** Запустить выполнение агента.
        *   **Обработка ответа:** Qwen-Agent `Assistant` вернет итератор. Последнее сообщение от ассистента будет содержать финальный ответ. Извлечь из него необходимую информацию (текст анализа, статус, уверенность). Формат ответа от LLM должен быть таким, чтобы эти части легко парсились (можно попросить LLM вернуть JSON в последнем сообщении).
        *   Вернуть структурированный результат.

4.  **Тестирование:**
    *   Написать простые тесты для агента, передавая ему моковые данные и проверяя, пытается ли он вызвать инструмент и как обрабатывает ответ.

## 3. Интеграция `QwenPensionAgent` в FastAPI Эндпоинт

**Цель:** Подключить разработанного агента к API для приема запросов и возврата результатов.

**Место:** Модификация `backend/app/main.py`.

**Шаги:**

1.  **Импорты:**
    *   Импортировать `QwenPensionAgent` и `CaseDataInput`.
2.  **Инициализация Агента:**
    *   Решить, будет ли агент создаваться один раз при старте приложения (в `lifespan` контекст-менеджере) или для каждого запроса. Для агентов с состоянием или тяжелой инициализацией предпочтительнее первый вариант.
    *   Пример для `lifespan`:
        ```python
        from backend.app.agents.qwen_pension_agent import QwenPensionAgent

        @asynccontextmanager
        async def lifespan(app: FastAPI):
            # ... (существующий код lifespan)
            print("Initializing QwenPensionAgent...")
            app.state.pension_agent = QwenPensionAgent() # Со всеми нужными конфигурациями
            print("QwenPensionAgent initialized.")
            yield
            # ... (существующий код lifespan для shutdown)
        ```
3.  **Модификация Эндпоинта (например, `/process`):**
    *   Изменить существующий эндпоинт или создать новый, который будет принимать `CaseDataInput` и, возможно, дополнительное текстовое описание.
    *   Внутри эндпоинта:
        *   Получить экземпляр агента (например, `agent = request.app.state.pension_agent`).
        *   Вызвать метод агента для анализа (например, `analysis_result = await agent.run_analysis(case_description_text="...", case_data_structured=case_data)`). `run_analysis` может потребоваться сделать асинхронным, если внутри него есть `await`.
        *   Вернуть результат, полученный от агента, в соответствующей Pydantic-модели ответа.
4.  **Обработка ошибок:** Добавить try-except блоки для отлова возможных ошибок при работе агента.

## 4. Адаптация RAG и Neo4j для Инструмента

**Цель:** Убедиться, что существующие RAG и Neo4j компоненты могут быть легко использованы `PensionKnowledgeRetrieverTool`.

**Место:** `backend/app/rag_core/*`, `backend/app/graph_builder.py`, `backend/app/graph_enricher.py`.

**Шаги:**

1.  **Рефакторинг `PensionRAG` (в `backend/app/rag_core/engine.py`):**
    *   Выделить основную логику поиска (векторный поиск, реранкинг) в отдельные методы, которые могут быть вызваны извне (из `PensionKnowledgeRetrieverTool`) с передачей поискового запроса и других необходимых параметров.
    *   Убедиться, что эти методы не зависят от состояния всего FastAPI приложения, если это возможно, или что необходимое состояние может быть передано.
2.  **Адаптация Neo4j-модулей (`graph_builder.py`, `graph_enricher.py`):**
    *   Предоставить четкие функции для:
        *   Извлечения сущностей из текста (если это делается с помощью Neo4j или нужно для запросов к нему).
        *   Запроса связанных данных для списка сущностей или статей.
        *   Форматирования результатов из Neo4j в удобный для инструмента вид.
    *   Эти функции будут вызываться из `PensionKnowledgeRetrieverTool` после получения результатов от LlamaIndex.

## 5. Промпт-Инжиниринг

**Цель:** Создать эффективные промпты для Qwen3, чтобы агент корректно выполнял свои задачи.

**Место:** В коде `QwenPensionAgent` (системный промпт) и, возможно, в конфигурации `PensionKnowledgeRetrieverTool` (если он использует LLM для суммаризации).

**Шаги:**

1.  **Разработка системного промпта для `QwenPensionAgent`:**
    *   Итеративно улучшать системный промпт, тестируя поведение агента.
    *   Включить:
        *   Роль и цель агента.
        *   Подробное описание `PensionKnowledgeRetrieverTool`: какие параметры принимает, что возвращает, когда его использовать.
        *   Инструкции по планированию действий (например, "Сначала проанализируй входные данные. Если информации недостаточно, используй `pension_knowledge_retriever` для получения данных из законодательства. Затем сформируй ответ...").
        *   Требования к формату ответа (структура JSON, обязательные поля: анализ, статус, уверенность, ссылки на статьи).
        *   (Важно!) Примеры few-shot диалогов или вызовов инструментов, чтобы показать желаемое поведение.
2.  **(Если применимо) Промпты для внутренних LLM-вызовов:** Если `PensionKnowledgeRetrieverTool` использует LLM для суммаризации, разработать для этого краткие и точные промпты.
